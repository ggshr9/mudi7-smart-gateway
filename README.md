# mudi7-smart-gateway

Bring-your-own-VPS smart proxy bootstrap for the **GL.iNet Mudi 7 (GL-E5800)** 5G travel router. After a factory reset, one script and a screen toggle gives you a three-state gateway:

| Screen state | Behavior |
| --- | --- |
| **VPN OFF** | Plain SIM-routed Wi-Fi. mihomo stopped, dnsmasq → public DNS. |
| **VPN ON + Global Mode** | Foreign traffic via VLESS-REALITY (Hysteria-2 fallback). CN traffic bypassed at the netfilter layer via a 4727-entry CIDR set. fake-IP DNS for transparent proxying. |
| **VPN ON + Policy Mode** | mihomo not started — reserved for whatever else you want (game accelerator, work VPN, etc.). |

The provisioning is driven entirely by a hotplug hook reacting to `wgclient1` ifup/ifdown, so the physical screen switch on the router is the only user-visible control.

## Why a fresh repo for this

GL.iNet ships a usable router, but turning it into a smart proxy gateway in China hits three traps that aren't documented anywhere I could find. Each of them cost hours, so this repo bakes the fixes in:

1. **CGNAT silently drops WireGuard return packets on well-known UDP ports.** Handshake init goes out, reply never makes it back through the carrier NAT. Random high port works instantly. The default in this script is `39753`.
2. **GL's policy framework DNAT-hijacks LAN DNS** (port 53 → port 2153 for any LAN query with the tunnel mark). The hijacker dnsmasq queries public DNS by default, returning REAL IPs and completely bypassing mihomo's fake-IP. The script reconfigures that second dnsmasq to use mihomo as upstream — durable, because fighting the DNAT itself is whack-a-mole.
3. **fw4 forward chain has no rule for the mihomo TUN device.** LAN → utun packets fall through to `handle_reject` and the client gets a TCP RST. Pernicious because `curl --resolve fake-IP` from the router itself works fine (OUTPUT chain, not FORWARD), so the failure only surfaces on real LAN devices. The script installs a `ruleset-post/utun-forward.nft` that inserts the missing accept rules at the chain head.

## Prerequisites

- A Mudi 7 (GL-E5800) in roughly factory-reset state, reachable at `192.168.8.1` over SSH or the GL Web UI.
- A VPS you control (any modern Linux box) running:
  - `sing-box` with VLESS+REALITY (TCP) and Hysteria-2 (UDP)
  - `wg-quick@wg0` on a random high UDP port (don't use 51820/51234!)
- Outbound network from Mudi → VPS that isn't being actively interfered with on those ports.

## Usage

```bash
git clone https://github.com/ggshr9/mudi7-smart-gateway.git
cd mudi7-smart-gateway

# Interactive wizard — prompts for VPS host, IP, VLESS/HY2 creds, etc.
# Generates mudi.env (mode 600). Re-run anytime to edit (existing values
# show as defaults). Or copy mudi.env.example and edit by hand if you prefer.
./setup.sh

# Push to Mudi:
scp mudi.env provision-mudi.sh root@192.168.8.1:/tmp/
ssh root@192.168.8.1 'cd /tmp && set -a && . ./mudi.env && set +a && ./provision-mudi.sh'
```

After the script finishes:

1. In GL Web UI → VPN → WireGuard Client → **Add Manually**, paste your `[Interface]` and `[Peer]` (endpoint = `${VPS_IP}:39753`, MTU 1408, GL generates the PrivateKey).
2. Click Apply — note the auto-generated Public Key.
3. Add that pubkey as a `[Peer]` in your VPS `wg0.conf`, `wg syncconf wg0 <(wg-quick strip wg0)`.
4. Set Mode = **Global** in the same dialog.
5. Toggle VPN ON on the screen. Verify `wg show wgclient1` shows a recent handshake.
6. Test from a LAN device: `baidu.com` direct, `google.com` via REALITY.

## Architecture (file layout after provisioning)

```
/usrdata/proxy/
  ├─ bin/mihomo                            mihomo binary
  └─ etc/config.yaml                       rendered config (fake-IP, VLESS, Hy2, PROXY group)

/etc/init.d/mihomo                         procd service (not enabled at boot)
/etc/hotplug.d/iface/99-vpn-mode           the brain — fires on wgclient1 ifup/ifdown

/usr/share/nftables.d/ruleset-post/
  ├─ utun-forward.nft                      ← the fix #3 above (insert rule)
  ├─ cn-bypass.nft                         CN IP set + prerouting mark
  └─ cn-cidr-elements.nft                  4727 CIDR entries, regenerated by cron

/usr/local/bin/update-cn-cidr.sh           daily refresh of the CIDR set
/etc/hosts                                 ← static pin for VPS_HOST (works around corporate DNS)
```

## What's *not* in this repo

- The VPS side (sing-box `config.json`, `wg0.conf`). You configure those once and they don't need a tool.
- The `mudi.env` file — that's where your secrets live. It's gitignored.
- Snapshots of pre-reset state used during development. Also gitignored.

## Troubleshooting

The provisioning installs a 5-minute health check (`/usr/local/bin/mudi-vpn-health.sh`) that watches for mihomo crashes, stale WG handshakes, missing TUN device, and dropped fw4 rules — and re-triggers the hook to recover. Most transient breakage self-heals within 5 minutes. Watch it with:

```bash
logread -e mudi-health | tail
```

If you're debugging a deeper problem, three checks pinpoint where things broke:

**1. Is WireGuard up?**

```bash
wg show wgclient1
```

- `latest handshake: X seconds ago` < 180s → tunnel healthy.
- `Unable to access interface: No such device` → screen toggle is OFF, or `ifup wgclient1` failed.
- Handshake stale > 3 min → carrier CGNAT timed out the mapping; the health check will re-up the interface.

**2. Is mihomo running and is the FORWARD chain in place?**

```bash
pgrep -x mihomo && echo alive || echo dead
ip link show utun 2>&1 | head -1
nft -a list chain inet fw4 forward | head -6
```

The forward chain MUST have both `iifname "br-lan" oifname "utun" accept` and the reverse at the head — without them, LAN devices get TCP RST. If missing, `fw4 reload` reapplies them from `/usr/share/nftables.d/ruleset-post/utun-forward.nft`.

**3. Is DNS returning fake-IPs (198.18.x.x)?**

```bash
nslookup google.com 127.0.0.1
```

Expected: `Address: 198.18.0.X`. If a real Google IP comes back, the wgclient1 dnsmasq (port 2153) lost its mihomo upstream — check `uci show dhcp.wgclient1.server`.

**Manually re-run the state-machine hook** (idempotent — safe to invoke any time):

```bash
INTERFACE=wgclient1 ACTION=ifup /etc/hotplug.d/iface/99-vpn-mode
logread -e vpn-mode | tail
```

**Tunables** live in `/etc/mudi-vpn.conf` (LAN_NET, TUN_GW, ports, etc.). Edit there and re-trigger the hook to apply — no need to re-run the whole provision script.

## Caveats / known limits

- Only tested on the **GL-E5800 / Mudi 7** running GL.iNet OpenWrt 23.05 with the Quectel SDK. Other GL hardware uses different fw4/ip-rule layouts and the FORWARD chain fix may need adjustment.
- China Mobile SIM cellular only — handshake stability through other CN carrier CGNATs is untested.
- The GL UI will show "Connecting…" if you toggle VPN ON via Web UI while the script is still bringing up the interface; ignore it and check `wg show wgclient1` for the real state.
